# 长任务执行清单

## 总目标

在用户睡觉期间连续推进信息安全大作业，不因单个步骤卡住而停止。优先跑通 WAM 全流程，使其达到题目要求中“大部分能复现”的程度；跑通后继续基于其他论文的明确思想进行创新优化，能做多少做多少，但不能凑数。

## 最高优先级规则

1. 不停工：完成当前任务后，必须回看本文档，选择下一项继续做。
2. 不问用户：遇到问题先自行判断、查日志、换方案、回滚或切换备选项目。
3. 不臆想创新：所有创新优化必须来自已经下载/阅读的相关论文思想，例如 DWSF、MBRS、TrustMark、EditGuard、CRMark、FIN、Robust-Wide、VINE 等。
4. 不凑数：每个创新必须有明确工程量、实验可验证性和报告可解释性。
5. 勤同步：每完成一个稳定阶段，必须：
   - 更新 `报告素材.md`
   - 写入 `实验记录/`
   - 必要时保存结果到 `结果输出/`
   - `git add` + `git commit`
6. 能回滚：如果某个创新或修改导致主流程坏掉，优先 `git status` 检查，再回滚自己本阶段的改动，保留上一个可运行版本。
7. 优先使用现有环境：先用已有 Python/conda 环境运行；仅在确定环境缺失且必要时安装轻量依赖。不要优先重装 PyTorch/CUDA 这类大组件。
8. 后台执行耗时任务：耗时下载、复现、批量实验应使用后台进程写日志；前台轮询日志、检查进度、决定下一步，避免被单个任务卡死。
9. 按题目要求推进：始终围绕 CCF-B 以上相关论文、复现/改进、实验对比、结果分析、代码截图和中文报告素材。

## 题目要求约束

- 方向需与课程内容相关：数字水印可对应数据安全、完整性保护、认证/溯源、攻击防御。
- 方案应基于 CCF-B 以上会议/期刊论文。
- 推荐路径：选择一篇主论文复现并改进创新，通过实验验证改进效果。
- 报告需要包含：
  - 背景和动机
  - 复现方案或改进方案原理
  - 实验验证和结果分析
  - 代码运行效果截图
  - 总结
- 代码需包含 README，最终和报告一起提交。

## 阶段 3：创新方案 A - DWSF 式多区域冗余嵌入与融合

来源论文：DWSF。

核心思想：

- 同一消息分散嵌入多个区域。
- 攻击后从多个幸存区域提取消息。
- 对多个区域结果进行融合，提升裁剪、遮挡、拼接后的恢复率。

任务：

1. 实现多区域重复嵌入。
2. 实现 majority vote 融合。
3. 实现 confidence weighted fusion。
4. 若可行，实现 similarity-based fusion。
5. 与原始 WAM 单区域方案在相同攻击上对比。
6. 做消融：
   - 单区域 WAM
   - 多区域不融合
   - 多区域 majority vote
   - 多区域 confidence fusion
   - 多区域 similarity fusion
7. 更新素材和实验记录。
8. git commit：`Add DWSF-style redundant regional fusion`

## 阶段 4：创新方案 B - MBRS 式真实/模拟 JPEG 鲁棒评测或增强

来源论文：MBRS。

核心思想：

- 使用真实 JPEG 与模拟/无攻击混合，提高 JPEG 鲁棒性。

可做方向：

- 先不重训模型，构造更完整 JPEG 攻击基线，比较不同质量因子。
- 若有余力，做轻量后处理策略：多尺度/多强度解码，选择置信度最高结果。

任务：

1. 实现 JPEG Q=95/85/75/65/50/30 批量评测。
2. 实现多次 JPEG + resize 组合攻击。
3. 尝试 multi-decode：对攻击图做若干预处理后检测，选择置信度最高消息。
4. 与 baseline 对比。
5. 更新素材和实验记录。
6. git commit：`Add MBRS-inspired JPEG robustness evaluation`

## 阶段 5：创新方案 C - TrustMark 式强度/质量权衡

来源论文：TrustMark。

核心思想：

- 通过强度参数控制不可感知性和鲁棒性权衡。
- 关注任意分辨率和工程化实用性。

可做方向：

1. 扫描 WAM 的 `scaling_w` 参数。
2. 记录不同 scaling_w 下：
   - PSNR / SSIM
   - bit accuracy
   - 攻击后恢复率
3. 选择 Pareto 较优参数。
4. 和 DWSF 多区域融合结合。
5. 更新素材和实验记录。
6. git commit：`Add TrustMark-inspired strength tradeoff study`

## 阶段 6：创新方案 D - EditGuard 式局部篡改安全场景

来源论文：EditGuard。

核心思想：

- 不只恢复 copyright bits，还定位被篡改区域。
- 将水印用于主动取证和内容完整性保护。

可做方向：

1. 构造局部篡改/遮挡/inpainting-like 攻击。
2. 利用 WAM detection mask 观察水印区域是否被破坏。
3. 将消息恢复结果和被破坏区域可视化。
4. 如果可行，输出“水印区域存活图”和“篡改影响分析”。
5. 更新素材和实验记录。
6. git commit：`Add EditGuard-inspired tamper localization analysis`

## 阶段 7：创新方案 E - CRMark/可逆思想参考

来源论文：CRMark。

核心思想：

- 关注鲁棒水印和图像恢复/可逆性。

可做方向：

- 如果 WAM 水印强度导致视觉质量下降，尝试降低强度并依赖多区域融合补偿鲁棒性。
- 形成“低扰动 + 多区域冗余”的设计解释：用空间冗余换取单点扰动降低。

任务：

1. 对比强水印单区域 vs 弱水印多区域。
2. 记录视觉质量和鲁棒性。
3. 更新素材和实验记录。
4. git commit：`Add low-distortion redundant watermark study`

如果上述阶段都完成且用户仍未醒，不要转去写报告、README 或做形式整理，继续围绕“读其他论文 - 提炼创新点 - 改主论文 - 实验验证”循环做实质工作。

阶段 8 的禁止事项：

- 不写正式报告正文。
- 不写 README 草稿。
- 不整理报告大纲。
- 不为了凑进度生成空泛图表或汇总表。
- 不做只有形式、没有新增论文依据或实验验证的任务。

持续循环任务：

1. 回到海选论文和已下载 Markdown，继续读与主论文相关的论文。
2. 找到明确可借鉴的创新点，必须能指出来源论文和对应思想。
3. 判断该创新是否能工程实现，不能只停留在概念层。
4. 将该创新落到主论文代码或实验流程中。
5. 跑实验验证该创新是否有正向效果，至少要有 baseline 对比。
6. 如果创新失败，记录失败原因；能修就修，不能修就回滚并换下一个创新点。
7. 每完成一个创新尝试，都同步 `报告素材.md`、实验记录和 git。

可继续挖掘的创新来源：

- DWSF：多区域分散嵌入、同步、消息融合。
- MBRS：真实/模拟 JPEG 混合、JPEG 鲁棒性增强。
- TrustMark：强度-质量权衡、任意分辨率工程化策略。
- EditGuard / OmniGuard：局部篡改、主动取证、版权信息和定位联合。
- CRMark：低扰动、可恢复/可逆思想。
- FIN / CIN：可逆结构、噪声层、强攻击下的解码稳健性。
- Robust-Wide / VINE：图像编辑攻击下的鲁棒性评测和增强。
